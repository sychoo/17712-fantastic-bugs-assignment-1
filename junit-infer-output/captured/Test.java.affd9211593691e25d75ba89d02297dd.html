10/**
11 * The <code>Test</code> annotation tells JUnit that the <code>public void</code> method
12 * to which it is attached can be run as a test case. To run the method,
13 * JUnit first constructs a fresh instance of the class then invokes the
14 * annotated method. Any exceptions thrown by the test will be reported
15 * by JUnit as a failure. If no exceptions are thrown, the test is assumed
16 * to have succeeded.
17 * <p>
18 * A simple test looks like this:
19 * <pre>
20 * public class Example {
21 * <b>&#064;Test</b>
22 * public void method() {
23 * org.junit.Assert.assertTrue( new ArrayList().isEmpty() );
24 * }
25 * }
26 * </pre>
27 * <p>
28 * The <code>Test</code> annotation supports two optional parameters for
29 * exception testing and for limiting test execution time.
30 *
31 * <h3>Exception Testing</h3>
32 * <p>
33 * The parameter <code>expected</code> declares that a test method should throw
34 * an exception. If it doesn't throw an exception or if it throws a different exception
35 * than the one declared, the test fails. For example, the following test succeeds:
36 * <pre>
37 * &#064;Test(<b>expected=IndexOutOfBoundsException.class</b>)
38 * public void outOfBounds() {
39 * new ArrayList&lt;Object&gt;().get(1);
40 * }
41 * </pre>
42 *
43 * Using the parameter <code>expected</code> for exception testing comes with
44 * some limitations: only the exception's type can be checked and it is not
45 * possible to precisely specify the code that throws the exception. Therefore
46 * JUnit 4 has improved its support for exception testing with
47 * {@link Assert#assertThrows(Class, ThrowingRunnable)} and the
48 * {@link org.junit.rules.ExpectedException ExpectedException} rule.
49 * With <code>assertThrows</code> the code that throws the exception can be
50 * precisely specified. If the exception's message or one of its properties
51 * should be verified, the <code>ExpectedException</code> rule can be used. Further
52 * information about exception testing can be found at the
53 * <a href="https://github.com/junit-team/junit4/wiki/Exception-testing">JUnit Wiki</a>.
54 *
55 * <h3>Timeout</h3>
56 * <p>
57 * The parameter <code>timeout</code> causes a test to fail if it takes
58 * longer than a specified amount of clock time (measured in milliseconds). The following test fails:
59 * <pre>
60 * &#064;Test(<b>timeout=100</b>)
61 * public void infinity() {
62 * while(true);
63 * }
64 * </pre>
65 * <b>Warning</b>: while <code>timeout</code> is useful to catch and terminate
66 * infinite loops, it should <em>not</em> be considered deterministic. The
67 * following test may or may not fail depending on how the operating system
68 * schedules threads:
69 * <pre>
70 * &#064;Test(<b>timeout=100</b>)
71 * public void sleep100() {
72 * Thread.sleep(100);
73 * }
74 * </pre>
75 * <b>THREAD SAFETY WARNING:</b> Test methods with a timeout parameter are run in a thread other than the
76 * thread which runs the fixture's @Before and @After methods. This may yield different behavior for
77 * code that is not thread safe when compared to the same test method without a timeout parameter.
78 * <b>Consider using the {@link org.junit.rules.Timeout} rule instead</b>, which ensures a test method is run on the
79 * same thread as the fixture's @Before and @After methods.
80 *
81 * @since 4.0
82 */
